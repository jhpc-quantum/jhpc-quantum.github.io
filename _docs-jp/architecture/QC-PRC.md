# 量子コンピュータ遠隔手続き呼び出し機構の設計

## QC request scheduler

RPCごとのQC request1は、QC request scheduler でスケジューリングされ、必要に応じて、QC hostでの実行のためのリクエストとして送信され、実行される。QCプログラムの実行は、量子コンピュータ本体にて実行されるが、その他のQCプログラムのコンパイルやトランスパイルは、量子コンピュータを管理するサーバーで行われる。

Simple clientで実行され、QCプログラムを１回しか実行しないjobを特にsimple QC jobと呼ぶ。HPC clientで実行されるQCプログラムの実行を含むjobをHPC QC jobと呼ぶ。

QC requestは、優先度を持ち、その優先度に従い実行される。Request のpriorityは、QC Client の実行環境により、QC sessionの情報に含まれる。例えば、HPC QC jobは、なるべくQCプログラムの実行待ち時間を避けるために高優先度で実行される。
 HPC QC jobのプログラムは、MPIで記述された複数ノードを用いる並列プログラムでもよい。QCプログラムの実行は通常マスターノードで行われる。マスタ以外のノードにおいても、ノードごとにQCプログラムの実行を可能にすることも検討する。

- 検討事項：マスタ以外のノードにおいても、ノードごとにQCプログラムの実行を可能にすることも検討する。

## HPCシステムと量子コンピュータのスケジューリング

Simple QC client は、clientプログラムがsingle nodeで、実行されるclientをいう。この場合は、client側のリソースは比較的安価であり、容易に停止・再開が可能である。現在、多く用いられているPythonのフレームワークで記述されたQC programのほとんどは、このsimple QC clientである。一方、HPC QC clientは、HPC（スパコン）で実行されるQC clientであり、多くの場合、複数のノードを用いるプログラムであり、プログラムを停止・再開することが難しい。そのため、QCプログラムの実行待ちのアイドル時間をさける必要がある。

HPC QC jobを効率的に実行するために、まずはHPCシステム（スパコン）が一つの場合を考える。通常、スパコンのジョブはスパコン上のジョブ管理システムによって管理されている。ユーザーは、スパコンでジョブを実行する場合は、ジョブ管理システム（ジョブスケジューラ）にジョブを投入（submit）して実行する。ジョブスケジューラは、投入されたジョブのリソースが確保され次第、ジョブを実行するようになっている。

このジョブスケジューラを利用して、HPC QC ジョブ用のスケジューラを用意する。このスケジューラでは、HPCシステム資源が確保されたジョブを一つだけ実行するように設定する。この実行されるHPC　QCジョブからのQCリクエストは、高優先度で実行する。

QC  request schedulerにおいて、QCプログラムの実行リクエストの実行時間は量子回路の実行時間ｘショット数となる。量子コンピュータの方式の違いにもよるが、実行時間は概ねでデコヒーレンスまでの時間であり、数ミリ秒となる。したがって、QCプログラムの実行リクエストの実行時間は数秒以内ということになる。一回の量子プログラムの実行が中断（preempt）できないとしても、ショット数を自動的に分割することにより、QCリクエストの実行時間を数秒以内になるように調整することとする。これにより、最優先度のQC実行リクエストは数秒以内に実行できるということになり、HPCシステムの待ち時間を最小化できる。なお、simple QC jobについても、ジョブ管理システムを導入し、ジョブキューで管理を行うが、こちらのジョブについては中断しても大きな資源の消費にはならないためにこのジョブからのQC実行リクエストは低優先度で実行され、高優先度のジョブのために実行が遅延する場合はリクエスト待ちでブロック状態になる。このsimple QC jobの場合は、同じノードを複数プロセスで利用してもよいので、このキューからの実行は複数jobを多重に実行してもよい。

また、対話的な利用をサポートする場合は、このjobの優先度をキューで管理するジョブよりも上げておくことで対話環境の利便性を向上できる可能性がある。さらに、実時間処理が必要なジョブがある場合など、様々な用途による効率的なスケジューリングの詳細については利用状況を見て、決定する。

- 検討事項：ジョブの種別による優先度や管理の方法の検討が必要。

## QCリクエストスケジューラの設計・実装

QCリクエストの効率的なスケジューリング方式はこれから検討することとするが、受け付けているQCプログラム実行リクエストから、適当なリクエストを選択し、バックエンド・スケジューラに投入する。その際、HPC QCジョブを効率的に実行するための優先度については配慮されるものとする。

 QC clientプログラムからQCリクエストスケジューラへのRPCには gRPC を用いるものとする。RPCは、非同期 API (gRPC による非同期 RPC ではなく同期 RPC を用い，1 RPC リクエストごとに 1 RPC リプライを返す/貰うまでブロックする RPC 方式  で実現する) であり，QCプログラム実行リクエストのRPCが成功した際にはリクエストID が返却され、以降のジョブの状態ポーリング，ジョブ結果取得のキーとして使用されるものとする．

### リクエストID の安全性
リクエスト ID は連番やPRC実行の日時その他アカウンティング情報が容易に推測できる構成を取らず，アカウンティング情報を含むがそれらが適切なフォーマットで結合されたバイト列に単方向ハッシュを施した一定長さのバイト列とする。

### RPC パラメタ詳細と RPC 通信路の安全性
RPC パラメタの詳細はこれからの検討事項となるが、認証のための token, ticket は既定で含まれるものとする。a) の認証を用いる場合には，その通信路が TLS 1.3 でデータ秘匿性およびデータ完全性が担保されなければならない が、b)の認証は富岳環境内部のみで使用され，それを利用するRPCの通信路は TCP でよい。

- 検討事項：富岳 login のための ssh 公開鍵設定用 pkcs#12 ファイルを有している富岳ユーザは，自宅やその他遠隔地からのジョブサブミッションにおいて，この TLS 1.3 通信確立にその pkcs#12 ファイル内の X.509 ユーザ証明書およびその秘密鍵を用いた TLS 1.3 クライアント認証が行え，その認証主体は富岳環境の uid であるため，token, ticket を用いずとも認証信頼度としては 1.1.1.b) と等価である．また，X.509 ユーザ証明書から生成したRFC 3820 Proxy Certificate による TLS 1.3 クライアント認証をサポートすれば，既存の HPCI-ID を持つユーザの uid が本システムの uid として使用可能になるが、これを採用するかは検討要。

### QCリクエスト処理のオーバーヘッド低減
QCリクエストのオーバーヘッド低減のため，HPC ジョブの投入のためのセッション (以下ジョブセッション) が生成でき，
ジョブセッション生成時に一度 b) の認証用 token, tiket による認証を受けた後は，その通信路をセッションとして以降ジョブセッションの破棄までの間，ジョブセッションを介した gRPC でジョブのサブミッション，状態ポーリング、結果取得が行えるように実装する。

QCリクエストスケジューラは，システムデーモンとして動作するアプリケーションとして一般に望まれる可用性と堅牢性を，その他のシステムとの連携を含めて付与可能であること．Linux sysremd, log rotation system 等との連携も含む。また、 運用の煩雑さの軽減のため，外部プロセス DBMS を用いずに実装する。 

- 検討事項：なんらかの理由でジョブセッションが (サーバ，クライアントのどちらかからでも) 破棄された時，そのセッションでサブミットしたジョブが終了していないなどの理由で再度セッションが必要になった場合，クライアントの都合でいつでも再度ジョブセッションを生成してもよいし，ジョブセッションを生成せず，逐一の gRPC に token, ticket を伴うアクセス要求を行っても良いこととする．

## 複数量子コンピュータと複数HPCシステムのQCジョブ・スケジューリング

構築を計画しているプラットフォームでは、複数のHPCシステム（スパコン）が、複数の量子コンピュータを共有する。

HPCシステムが一つの場合にはHPCシステムのスケジューラにより、高優先度で実行するHPC QC jobを制限できる。しかし、複数のHPCシステムがある場合には、それぞれのHPCシステムから高優先度のHPC QCジョブが投入されることになり、それぞれのHPC QC　ジョブが干渉した場合には効率が低下することが考えられる。

これを避けるためには、異なるHPCスケジューラ間で実行されるHPC QCジョブが１つになるように調整するための仕組みが必要となる。この調整機能は、複数HPCシステムのスケジューラ間で連携して、あるHPCシステムのHPC　QCジョブが実行されているときには、他方のHPCシステムのジョブスケジューラにおいてはジョブの実行を保留する機能を実装する必要がある。これを、ジョブスケジューラ連携機構、と呼ぶことにする。

実装の方式については、いくつかの選択肢があるが、量子コンピュータごとに実行開始の権限を管理するプロセスをHPC　QCジョブスケジューラが到達可能なサーバー（おそらく、管理対象の量子コンピュータのQCリクエストスケジューラと同じサーバー）で動作させ、このプロセスによりHPC QCジョブの実行権限を管理することが考えられる。

複数のQC hostを同時に用いるQC Clientはプログラミング上は可能であるが、HPC QC clientの場合は複数の量子コンピュータの優先度実行を行わなくてはならないため、複数の量子コンピュータの同じジョブからの同時利用は行わないものとする。しかし、simple QC clientの場合は、client側のリソースに大きな影響がないため、複数の量子コンピュータを一つのジョブから同時に利用することが可能である。

- 検討事項：複数の量子コンピュータ（シミュレータを含む）を同時利用するアプリケーションはあるのか？

- 検討事項：富岳および東大・阪大のスパコンにおいて、量子HPC専用のキューとジョブスケジューラ連携機構を動作してもらえるか、は今後の課題。

## 量子コンピュータ・バックエンド・インターフェス
量子プログラムの実行は，実際の量子コンピュータの動作を制御する制御マシンに対して各種操作指示を与えることで実現される。

量子コンピュータベンダーから提供されるスケジューラをバックエンド・スケジューラと呼ぶことにする。フロントエンド・スケジューラであるQCリクエストスケジューラから、量子コンピュータごとに決められたAPIを用いてQCプログラムを実行する。

- IBMの場合は、REST APIで提供される。
- Quantinuumの場合は、gRPCで提供される（予定）

このAPIに用いて、量子コンピュータごとの特性に応じた効率的な実行を可能とする順序あるいは手順でバックエンド・スケジューラに指示を与える。

- 検討事項：各量子コンピュータに応じたQCリクエストの実行方式の詳細については、ベンダーとの協議による検討が必要。また、効率的な実行に必要なフロントエンドへのQCリクエストに付加される情報についても検討。
